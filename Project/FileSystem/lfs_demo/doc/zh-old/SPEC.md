## littlefs技术规格

这是小文件系统的技术规范。这个文件涵盖了如何将littlefs存储在磁盘上的技术细节，以用于内省和工具。本文档假设您熟悉littlefs的设计，有关littlefs如何工作的更多信息，请查看出[DESIGN.md]（DESIGN.md）。

```
   | | |     .---._____
  .-----.   |          |
--|o    |---| littlefs |
--|     |---|          |
  '-----'   '----------'
   | | |
```

## 一些快速笔记

- littlefs是基于块的文件系统。磁盘分为一个数组
      大小均匀的块，用作存储的逻辑单元。

- 块指针以32位存储，具有特殊值`0xffffffff`
      代表一个空的块地址。

- 除了逻辑块大小（通常与擦除匹配）
      块大小），littlefs还使用程序块大小和读取块大小。
      这些确定块设备操作的对齐方式，但不需要
      保持一致的可移植性。

- 默认情况下，littlefs中的所有值均以little-endian字节顺序存储。

## 目录/元数据对

元数据对构成littlefs的骨干，并为分布式原子更新。甚至超级块也存储在元数据对中。

顾名思义，元数据对存储在两个块中，其中一个块在掉电期间提供备份，以防断电。这两个块不一定是顺序的，并且可能在磁盘上的任何位置，因此指向磁盘的“指针”元数据对存储为两个块指针。

最重要的是，每个元数据块的行为都像一个可附加的日志，其中包含一个
可变的提交次数。可以将提交追加到元数据登录中为了更新元数据而无需擦除周期。注意连续的提交可能会取代先前提交中的元数据。只有最新的元数据应被视为有效。

元数据块的高级布局非常简单：

```
  .---------------------------------------.
.-|  revision count   |      entries      |  \
| |-------------------+                   |  |
| |                                       |  |
| |                                       |  +-- 1st commit
| |                                       |  |
| |                   +-------------------|  |
| |                   |        CRC        |  /
| |-------------------+-------------------|
| |                entries                |  \
| |                                       |  |
| |                                       |  +-- 2nd commit
| |    +-------------------+--------------|  |
| |    |        CRC        |    padding   |  /
| |----+-------------------+--------------|
| |                entries                |  \
| |                                       |  |
| |                                       |  +-- 3rd commit
| |         +-------------------+---------|  |
| |         |        CRC        |         |  /
| |---------+-------------------+         |
| |           unwritten storage           |  more commits
| |                                       |       |
| |                                       |       v
| |                                       |
| |                                       |
| '---------------------------------------'
'---------------------------------------'
```

每个元数据块包含一个32位修订版计数，后跟多个提交。每个提交都包含可变数量的元数据条目的32位CRC

还要注意，条目不一定要字对齐。这使我们能够更紧凑地存储元数据，但是我们只能写到我们的程序块大小对齐的地址这中。这意味每个提交都可能为了对齐而做填充。

元数据块字段：

1. **修订计数（32位）** - 每个擦除周期递增。如果两个块
   包含有效的提交，仅包含具有最新修订计数的块
   应该使用。必须使用序列比较来避免出现问题
   整数溢出。

2. **CRC（32位）**- 检测掉电或其他写入引起的损坏
   问题。使用初始化为多项式0x04c11db7的CRC-32
   与`0xffffffff`。

条目本身存储为32位标记，后跟可变长度数据块，但是准确地存储这些标签的方式有些棘手。

元数据块支持正向和反向迭代。为了在不复制每个标签空间的情况下，相邻条目具有标签从“ 0xffffffff”开始异或在一起。

```
 Forward iteration                        Backward iteration

.-------------------.  0xffffffff        .-------------------.
|  revision count   |      |             |  revision count   |
|-------------------|      v             |-------------------|
|      tag ~A       |---> xor -> tag A   |      tag ~A       |---> xor -> 0xffffffff
|-------------------|      |             |-------------------|      ^
|       data A      |      |             |       data A      |      |
|                   |      |             |                   |      |
|                   |      |             |                   |      |
|-------------------|      v             |-------------------|      |
|      tag AxB      |---> xor -> tag B   |      tag AxB      |---> xor -> tag A
|-------------------|      |             |-------------------|      ^
|       data B      |      |             |       data B      |      |
|                   |      |             |                   |      |
|                   |      |             |                   |      |
|-------------------|      v             |-------------------|      |
|      tag BxC      |---> xor -> tag C   |      tag BxC      |---> xor -> tag B
|-------------------|                    |-------------------|      ^
|       data C      |                    |       data C      |      |
|                   |                    |                   |    tag C
|                   |                    |                   |
|                   |                    |                   |
'-------------------'                    '-------------------'
```

在深入了解标签编码之前，需要注意的最后一件事。每个标签包含一个有效位，用于指示该标签及其所包含的提交是否为有效的。该有效位是在标记和提交中找到的第一位，并且可以用来告诉我们是否尝试写入块。

```
| |-------------------+                   |    |   |

| |-------------------+-------------------|   ||
```

## 元数据标签

因此，在littlefs中，32位标记描述了每种类型的元数据。这意味着每种类型的元数据，包括文件条目，目录字段和全局状态。甚至用于标记提交结束的CRC都有自己的标签。

因此，标签格式包含一些密集打包的信息。

备注：有多种类型的类型可以分解为更多信息：

```
[----            32             ----]
[1|--  11   --|--  10  --|--  10  --]
 ^.     ^     .     ^          ^- length
 |.     |     .     '------------ id
 |.     '-----.------------------ type (type3)
 '.-----------.------------------ valid bit
  [-3-|-- 8 --]
    ^     ^- chunk
    '------- type (type1)
```

在继续之前，需要注意一件事。这些标签是**不**存储在little-endian中。实际存储在提交中的标签在big-endian中（这是在big-endian中存储的littlefs中唯一的内容）。这有点疯狂是因为有效位必须是提交的第一位，当转换为little-endian时，有效位会找到本身位于字节4中。我们可以重组标签以将有效位存储在较低的位置，但是，因为所有字段都不是字节对齐的，所以这会更多不仅仅是将标签存储在big-endian中。

另外要注意的是，标签“ 0x00000000”和“ 0xffffffff”都是
无效，可用于空值。

元数据标签字段：

1. **有效位（1位）**-指示标签是否有效。
2. **Type3（11位）**-标签的类型。该字段进一步细分
   转换为3位抽象类型和8位块字段。注意值
   0x000无效，未分配类型。
3. **Type1（3位）**-标签的抽象类型。将标签分组为
   8个有助于进行位掩码查找的类别。
4. **块（8位）**-块字段用于不同目的
   抽象类型。type1 + chunk + id为广告代码中的每个标签形成唯一的标识符
   元数据块。
5. **Id（10位）** -与标签关联的文件ID。元数据中的每个文件
   块获得一个唯一的ID，该ID用于将标签与该文件相关联。这
   特殊值“ 0x3ff”用于与广告代码不相关的任何标签
   文件，例如目录和全局元数据。
6. **长度（10位）**-数据长度，以字节为单位。特殊价值
   “ 0x3ff”表示该标签已被删除。

## 元数据类型

以下是littlefs中元数据的详尽列表。

---
#### `0x401` LFS_TYPE_CREATE

使用此ID创建一个新文件。请注意，元数据块中的文件
不一定需要创建标签。创建所做的一切就是移到任何
使用此ID的文件。从这个意义上说，创建类似于插入
虚构的文件数组。

创建和删除标签允许littlefs将文件保留在目录中
按文件名的字母顺序排序。

---
#### `0x4ff` LFS_TYPE_DELETE

删除具有此ID的文件。逆向创建，此标记移至
与此id相邻的任何文件都类似于从虚构中删除的文件
文件数组。

---
#### `0x0xx` LFS_TYPE_NAME

将ID与文件名和文件类型相关联。

数据包含以ASCII字符串存储的文件名（可以扩展为
未来的UTF8）。

此标记中的块字段指示8位文件类型，可以是以下一种
以下。

当前，名称标签必须位于与ID和ID相关联的任何其他标签之前
在不删除文件的情况下无法重新分配。

名称标签的布局：

名称字段：

1. **文件类型（8位）**-文件的类型。

2. **文件名**-以ASCII字符串存储的文件名。

#### `0x0ff` LFS_TYPE_SUPERBLOCK

将id初始化为超级块条目。

超级块条目是用于存储格式时间配置的特殊条目
并识别文件系统。

这个名字有点用词不当。虽然超级块条目具有相同的功能
目的是在其他文件系统中找到的超级块，在littlefs中
没有得到一个专用的块。相反，超级块条目是重复的
跨越以块0和1为根的元数据对的链接列表。
元数据对加倍作为文件系统的根目录。

```
 .--------.  .--------.  .--------.  .--------.  .--------.
.| super  |->| super  |->| super  |->| super  |->| file B |
|| block  | || block  | || block  | || block  | || file C |
||        | ||        | ||        | || file A | || file D |
|'--------' |'--------' |'--------' |'--------' |'--------'
'--------'  '--------'  '--------'  '--------'  '--------'

\----------------+----------------/ \----------+----------/
          superblock pairs               root directory
```

文件系统仅从根目录开始。超级块元数据
根对每次压实时都会增加对，以延长
设备的使用寿命成倍增长。

超级块条目的内容与
超级块类型和内联结构标签。名称标签包含魔术
字符串“ littlefs”，而inline-struct标记包含版本和
配置信息。

超级块名称标签和内联结构标签的布局：

```
        tag                          data
[--      32      --][--      32      --|--      32      --]
[1|- 11 -| 10 | 10 ][---              64               ---]
 ^    ^     ^    ^- size (8)           ^- magic string ("littlefs")
 |    |     '------ id (0)
 |    '------------ type (0x0ff)
 '----------------- valid bit

        tag                          data
[--      32      --][--      32      --|--      32      --|--      32      --]
[1|- 11 -| 10 | 10 ][--      32      --|--      32      --|--      32      --]
 ^    ^     ^    ^            ^- version         ^- block size      ^- block count
 |    |     |    |  [--      32      --|--      32      --|--      32      --]
 |    |     |    |  [--      32      --|--      32      --|--      32      --]
```

超级块字段：

1. **魔力字符串（8字节）**-指示存在魔力的魔力字符串
   设备上的littlefs。必须为字符串“ littlefs”。

2. **版本（32位）**-格式化时littlefs的版本。版本号
   以32位值编码，高16位包含主
   版本，以及包含次要版本的低16位。

   该规范描述了版本2.0（`0x00020000`）。

3. **块大小（32位）**-块使用的逻辑块大小
   文件系统（以字节为单位）。

4. **块计数（32位）**-文件系统中的块数。

5. **Name max（32位）**-文件名的最大大小（以字节为单位）。

6. **文件最大（32位）**-文件的最大大小（以字节为单位）。

7. **Attr max（32位）**-文件属性的最大大小，以字节为单位。

超级块也必须始终是元数据对中的第一个条目（id 0）
作为写入该块的第一个条目。这意味着超级块
可以仅使用偏移量从设备读取条目。

#### `0x6xx` LFS_TYPE_TAIL

提供元数据对本身的尾部指针。

元数据对的尾部指针在littlefs中用于链接列表
包含所有元数据对。块字段包含尾巴的类型，
指示以下元数据对是否是目录的一部分
（硬尾）或仅用于遍历文件系统（软尾）。

```
         .--------.
        .| dir A  |-.
        ||softtail| |
.--------|        |-'
|       |'--------'
|       '---|--|-'
|        .-'    '-------------.
|       v                      v
|  .--------.  .--------.  .--------.
'->| dir B  |->| dir B  |->| dir C  |
  ||hardtail| ||softtail| ||        |
  ||        | ||        | ||        |
  |'--------' |'--------' |'--------'
  '--------'  '--------'  '--------'
```

目前，任何类型都可以取代元数据对中的所有其他前面的尾部，
但是如果添加了其他元数据对状态，则可能会更改。

关于元数据对链接列表的说明：通常，此链接列表包含
文件系统中的每个元数据对。但是，有些操作
