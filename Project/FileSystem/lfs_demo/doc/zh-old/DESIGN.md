## littlefs的设计

专为微控制器设计的故障保护文件系统。

```
   | | |     .---._____
  .-----.   |          |
--|o    |---| littlefs |
--|     |---|          |
  '-----'   '----------'
   | | |
```

littlefs最初是作为实验来学习文件系统设计的
在微控制器的上下文中。问题是：您将如何建立一个
无需使用即可抵御功耗和闪存磨损的文件系统
无限的记忆？

本文档涵盖了littlefs的高级设计，它有何不同
而不是其他文件系统，以及使我们到这里的设计决策。为了
有关磁盘上每个位的低级详细信息，请查看[SPEC.md]（SPEC.md）。

## 问题

嵌入式系统littlefs的目标通常是具有以下功能的32位微控制器：
大约32 KiB的RAM和512 KiB的ROM。这些通常与SPI NOR配对
具有约4 MiB闪存存储空间的闪存芯片。这些设备太小，无法
Linux和大多数现有文件系统，需要使用以下代码专门编写
大小要紧。

Flash本身是一项有趣的技术，它有自己的怪癖和
细微差别。与其他形式的存储不同，写入闪存需要两个
操作：擦除和编程。编程（将位设置为0）为
相对便宜，并且可以非常精细。但是擦除（将位设置为1），
需要昂贵且具有破坏性的操作，因此必须为其命名。
[Wikipedia] [wikipedia-flash]提供了有关Flash的工作原理的更多信息。

为了使情况更加烦人，这些嵌入式程序非常常见
系统随时会断电。通常，微控制器代码很简单，
反应性的，没有关闭例程的概念。这是一个很大的挑战
用于永久性存储，不幸的断电会损坏存储，
使设备无法恢复。

这就给我们带来了嵌入式文件系统的三个主要要求。

1. **功耗恢复能力**-在这些系统上，随时可能断电。
   如果断电破坏了任何持久性数据结构，则可能导致
   设备变得无法恢复。嵌入式文件系统必须设计为
   在任何写操作期间都可从断电中恢复。

1. **损耗均衡**-写入闪存具有破坏性。如果是文件系统
   重复写入同一块，最终该块将耗尽。
   不考虑磨损的文件系统很容易烧穿块
   用于存储经常更新的元数据并导致设备过早死亡。

1. **有限的RAM / ROM **-如果上述要求还不够，这些
   系统的内存量也非常有限。这可以防止许多
   现有的文件系统设计，可以依靠相对大量的
   用于临时存储文件系统元数据的RAM。

   对于ROM，这意味着我们需要保持设计简单并重用代码路径
   可能的。对于RAM，我们有一个更严格的要求，所有RAM使用情况是
   有界。这意味着RAM使用率不会随着文件系统中的更改而增加
   文件的大小或数量。这可能会带来独特的挑战
   简单的操作（例如遍历文件系统）变得令人惊讶
   难的。

## 现有设计？

那么，那里已经有什么了？当然，有许多不同的文件系统，
但是，他们经常彼此共享和借鉴功能。如果我们看
功率损耗弹性和磨损平衡，我们可以将这些范围缩小到少数
设计。

1.首先，我们具有非弹性的基于块的文件系统，例如[FAT]和
   [ext2]。这些是最早的文件系统设计，通常是最简单的。
   在这里，存储分为几个块，每个文件都存储在一个文件中。
   块的集合。如果不进行修改，这些文件系统就不会
   断电恢复能力强，因此更新文件就像重写块一样简单
   到位。

   ```
            .--------.
            |  root  |
            |        |
            |        |
            '--------'
            .-'    '-.
           v          v
      .--------.  .--------.
      |   A    |  |   B    |
      |        |  |        |
      |        |  |        |
      '--------'  '--------'
      .-'         .-'    '-.
     v           v          v
.--------.  .--------.  .--------.
|   C    |  |   D    |  |   E    |
|        |  |        |  |        |
|        |  |        |  |        |
'--------'  '--------'  '--------'
   ```

   由于它们的简单性，这些文件系统通常都是最快的
   和最小的。但是，缺乏电源弹性的能力并不强，并且
   存储位置和数据的绑定关系删除了文件系统的
   处理磨损的能力。

2.在完全不同的方向上，我们有日志文件系统，例如
   [JFFS]，[YAFFS]和[SPIFFS]，存储位置未绑定到
   数据，而是将整个存储空间用于循环日志，即
   附加了对文件系统所做的所有更改。写作追加了新内容
   进行更改，而读取则需要遍历日志以重建文件。
   一些日志记录文件系统会缓存文件以避免读取成本，但这是需要
   在RAM的权衡下。

   ```
                                                          v
.--------.--------.--------.--------.--------.--------.--------.--------.
|        C        | new B  | new A  |                 |   A    |   B    |
|                 |        |        |->               |        |        |
|                 |        |        |                 |        |        |
'--------'--------'--------'--------'--------'--------'--------'--------'
   ```

   日志文件系统非常漂亮。有了校验和，我们可以轻松地
   通过忽略故障来检测功率损耗并退回到先前的状态
   追加。如果那还不够好，那么它们的循环特性就意味着
   日志文件系统完美地将磨损分配到整个存储中。

   主要缺点是性能。如果我们看垃圾收集，
   从日志末尾清除过时的数据的过程，我还没有
   看到没有两个成本之一的纯日志文件系统：

   1. _O（n＆sup2;）_运行时
   2. _O（n）_ RAM

   SPIFFS在这里是一个非常有趣的情况，因为它使用了重复
   NOR Flash的程序既是原子的又是屏蔽的。这很整齐
   解决方案，但是它限制了您可以支持的存储类型。

3.也许最常见的文件系统类型是日记文件系统。
   当您将基于块的文件系统与日志记录配对时发生的后代
   文件系统。[ext4]和[NTFS]是很好的例子。在这里，我们以正常
   基于块的文件系统，并在其中记录所有更改的地方添加有界日志
   在它发生之前。

   ```
                          journal
                         .--------.--------.
            .--------.   | C'| D'|     | E'|
            |  root  |-->|   |   |->   |   |
            |        |   |   |   |     |   |
            |        |   '--------'--------'
            '--------'
            .-'    '-.
           v          v
      .--------.  .--------.
      |   A    |  |   B    |
      |        |  |        |
      |        |  |        |
      '--------'  '--------'
      .-'         .-'    '-.
     v           v          v
.--------.  .--------.  .--------.
|   C    |  |   D    |  |   E    |
|        |  |        |  |        |
|        |  |        |  |        |
'--------'  '--------'  '--------'
   ```


   这种文件系统从两个方面都发挥了最大的作用。性能可以
   与基于块的文件系统一样快（尽管更新日记确实具有
   费用较低），并且对日志的原子更新允许文件系统执行以下操作：
   在断电的情况下恢复。

   不幸的是，日记文件系统有两个问题。他们是
   相当复杂，因为实际上有两个文件系统在运行
   并行，这要付出代码大小的代价。他们也没有提供任何保护
   防磨损，因为存储位置之间有很强的关系
   和数据。

4.最后但并非最不重要的一点是，我们拥有写时复制（COW）文件系统，例如
   [btrfs]和[ZFS]。这些与其他基于块的文件系统非常相似，
   但是所有更新都是通过创建来执行的，而不是就地更新块
   具有更改的副本，并用替换对旧块的任何引用
   我们的新街区。递归地将我们所有的问题向上推，直到我们
   到达文件系统的根目录，该目录通常存储在很小的日志中。

   ```
            .--------.                  .--------.
            |  root  |       write      |new root|
            |        |        ==>       |        |
            |        |                  |        |
            '--------'                  '--------'
            .-'    '-.                    |    '-.
           |  .-------|------------------'        v
           v v        v                       .--------.
      .--------.  .--------.                  | new B  |
      |   A    |  |   B    |                  |        |
      |        |  |        |                  |        |
      |        |  |        |                  '--------'
      '--------'  '--------'                  .-'    |
      .-'         .-'    '-.    .------------|------'
     |           |          |  |             v
     v           v          v  v        .--------.
.--------.  .--------.  .--------.      | new D  |
|   C    |  |   D    |  |   E    |      |        |
|        |  |        |  |        |      |        |
|        |  |        |  |        |      '--------'
'--------'  '--------'  '--------'
   ```

   COW文件系统很有趣。它们提供的性能与
   基于块的文件系统，同时设法在不进行原子更新的情况下进行
   将数据更改直接存储在日志中。他们甚至取消了存储的关联
   数据的位置，这为损耗平衡​​创造了机会。

   好吧，差不多。无限制的更新向上运动会导致一些
   问题。因为对COW文件系统的更新直到完成才停止
   到达根目录，更新可以级联成比
   原始数据将需要。最重要的是，向上运动
   将这些写入集中到该块中，该块可能比用尽的时间更早
   文件系统的其余部分。

## littlefs

那么littlefs是做什么的呢？

如果我们看一下现有的文件系统，有两种有趣的设计模式
脱颖而出，但每个人都有自己的一系列问题。记录，其中
提供独立的原子性，运行时性能较差。和COW数据
性能良好的结构将原子性问题向上推。

我们可以解决这些限制吗？

考虑记录。它具有_O（n＆sup2;）_运行时或_O（n）_ RAM成本。我们
不能避免这些费用，但是，如果我们将上限设置为
至少可以防止理论成本成为问题。这取决于
超级秘密的计算机科学技巧，您可以在其中伪装任何算法
通过限制输入，复杂度为_O（1）_。

对于COW数据结构，我们可以尝试稍微扭曲一下定义。
假设我们的COW结构不会在一次写入后复制，而是
_n_写后复制。这不会更改大多数COW属性（假设您
可以原子地写！），但是它的作用是防止
穿。这种边界写入复制（CObW）仍然着重于磨损，但在
在每个级别上，我们将磨损的传播程度除以_n_。有足够的
大的（n>分支因数）磨损传播不再是问题。

看到这是怎么回事？单独，日志记录和COW是不完善的解决方案，
具有限制其实用性的弱点。但是，如果我们将两者合并，他们可以
相互解决彼此的局限。

这就是littlefs背后的想法。在子块级别，构建了littlefs
小型，两个块日志中的任何一个，这些日志提供原子更新到任何地方的元数据
在文件系统上。在超级块级别，littlefs是块的CObW树
可以按需驱逐。

```
                    root
                   .--------.--------.
                   | A'| B'|         |
                   |   |   |->       |
                   |   |   |         |
                   '--------'--------'
                .----'   '--------------.
       A       v                 B       v
      .--------.--------.       .--------.--------.
      | C'| D'|         |       | E'|new|         |
      |   |   |->       |       |   | E'|->       |
      |   |   |         |       |   |   |         |
      '--------'--------'       '--------'--------'
      .-'   '--.                  |   '------------------.
     v          v              .-'                        v
.--------.  .--------.        v                       .--------.
|   C    |  |   D    |   .--------.       write       | new E  |
|        |  |        |   |   E    |        ==>        |        |
|        |  |        |   |        |                   |        |
'--------'  '--------'   |        |                   '--------'
                         '--------'                   .-'    |
                         .-'    '-.    .-------------|------'
                        v          v  v              v
                   .--------.  .--------.       .--------.
                   |   F    |  |   G    |       | new F  |
                   |        |  |        |       |        |
                   |        |  |        |       |        |
                   '--------'  '--------'       '--------'
```

仍然存在一些小问题。小原木在以下方面可能会很昂贵
存储，在最坏的情况下，一小条日志的成本是原始数据大小的4倍。
由于分配发生，CObW结构需要有效的块分配器
每个_n_写。保持RAM使用率仍然是一个挑战
不变。

##元数据对

元数据对是littlefs的骨干。这些是两个小块的日志
允许在文件系统中的任何地方进行原子更新。

为什么要两个街区？好吧，日志是通过将条目附加到循环缓冲区中来工作的
存储在磁盘上。但是请记住，闪存的写入粒度有限。我们可以
将新数据增量编程到已擦除的块上，但是我们需要擦除完整的数据
一次封锁。这意味着，为了使我们的循环缓冲区起作用，我们
需要一个以上的块。

我们可以将日志扩大到两个以上，但是下一个挑战是如何
我们是否存储对这些日志的引用？因为块本身被擦除
在写期间，使用数据结构跟踪这些块很复杂。
这里的简单解决方案是为每个元数据存储两个块地址
一对。这样做还有一个好处，就是我们可以在
元数据对是独立的，并且我们不会降低
其他操作。

为了确定哪个元数据块是最新的，我们存储了
我们使用[序列算术]比较的修订计数[wikipedia-sna]
（非常方便，可以避免整数溢出问题）。方便地，这
修订计数还使我们大致了解发生了多少次擦除
块。

```
metadata pair pointer: {block 0, block 1}
                           |        '--------------------.
                            '-.                           |
disk                           v                          v
.--------.--------.--------.--------.--------.--------.--------.--------.
|                 |        |metadata|                 |metadata|        |
|                 |        |block 0 |                 |block 1 |        |
|                 |        |        |                 |        |        |
'--------'--------'--------'--------'--------'--------'--------'--------'
                               '--.                  .----'
                                   v                v
             metadata pair .----------------.----------------.
                           |   revision 11  |   revision 12  |
             block 1 is    |----------------|----------------|
             most recent   |       A        |       A''      |
                           |----------------|----------------|
                           |    checksum    |    checksum    |
                           |----------------|----------------|
                           |       B        |       A'''     | <- most recent A
                           |----------------|----------------|
                           |       A''      |    checksum    |
                           |----------------|----------------|
                           |    checksum    |       |        |
                           |----------------|       v        |
                           '----------------'----------------'
```

那么，我们如何原子地更新我们的元数据对呢？原子性（一种
功率损耗弹性）需要两个部分：冗余和错误检测。
错误检测可以提供校验和，在littlefs的情况下，
使用32位[CRC] [wikipedia-crc]。另一方面，保持冗余，
需要多个阶段。

1.如果我们的程序块未满，并且程序大小足够小，可以让我们
   追加更多条目，我们可以简单地将条目追加到日志中。因为
   我们不会覆盖原始条目（请记住，重写Flash需要
   删除），如果在断开过程中断电，我们仍然拥有原始条目
   附加。

   ```
                                 commit A
.----------------.----------------.    .----------------.----------------.
|   revision 1   |   revision 0   | => |   revision 1   |   revision 0   |
|----------------|----------------|    |----------------|----------------|
|       |        |                |    |       A        |                |
|       v        |                |    |----------------|                |
|                |                |    |    checksum    |                |
|                |                |    |----------------|                |
|                |                |    |       |        |                |
|                |                |    |       v        |                |
|                |                |    |                |                |
|                |                |    |                |                |
|                |                |    |                |                |
|                |                |    |                |                |
'----------------'----------------'    '----------------'----------------'
   ```

   请注意，littlefs不会为每个条目维护一个校验和。许多伐木
   文件系统可以做到这一点，但是它限制了您可以在单个原子中更新的内容
   手术。我们可以做的是将多个条目分组为一个提交
   共享一个校验和。这使我们可以更新多个不相关的片段
   只要它们位于相同的元数据对上即可。

   ```
                              commit B and A'
.----------------.----------------.    .----------------.----------------.
|   revision 1   |   revision 0   | => |   revision 1   |   revision 0   |
|----------------|----------------|    |----------------|----------------|
|       A        |                |    |       A        |                |
|----------------|                |    |----------------|                |
|    checksum    |                |    |    checksum    |                |
|----------------|                |    |----------------|                |
|       |        |                |    |       B        |                |
|       v        |                |    |----------------|                |
|                |                |    |       A'       |                |
|                |                |    |----------------|                |
|                |                |    |    checksum    |                |
|                |                |    |----------------|                |
'----------------'----------------'    '----------------'----------------'
   ```

2.如果我们的块_is_充满了条目，则需要以某种方式删除过时的
   条目为新的条目腾出空间。这个过程叫做垃圾
   收集，但是因为littlefs有多个垃圾收集器，所以我们
   也称为这种特定情况下的压缩。

   与其他文件系统相比，littlefs的垃圾收集器相对而言
   简单的。我们想要避免RAM消耗，所以我们使用了一种蛮力
   解决方案，我们检查每个条目的位置，以查看是否有较新的条目
   书面。如果该条目是最新条目，则将其添加到新块中。这
   是有两个障碍变得很重要的地方，如果我们失去动力，我们仍然有
   我们原始块中的所有内容。

   在此压缩步骤中，我们还将擦除元数据块并递增
   修订计数。因为我们可以一次提交多个条目，所以我们可以
   将所有这些更改写入第二个块，而无需担心功耗
   失利。只有在写入提交的校验和时，才进行压缩
   条目和修订计数将被提交并可读。

   ```
                        commit B', need to compact
.----------------.----------------.    .----------------.----------------.
|   revision 1   |   revision 0   | => |   revision 1   |   revision 2   |
|----------------|----------------|    |----------------|----------------|
|       A        |                |    |       A        |       A'       |
|----------------|                |    |----------------|----------------|
|    checksum    |                |    |    checksum    |       B'       |
|----------------|                |    |----------------|----------------|
|       B        |                |    |       B        |    checksum    |
|----------------|                |    |----------------|----------------|
|       A'       |                |    |       A'       |       |        |
|----------------|                |    |----------------|       v        |
|    checksum    |                |    |    checksum    |                |
|----------------|                |    |----------------|                |
'----------------'----------------'    '----------------'----------------'
   ```

3.如果我们的块中充满了所有条目_and_我们找不到任何垃圾，那又如何呢？
   在这一点上，大多数日志文件系统将返回一个错误，指示没有
   有更多可用空间，但是因为我们的日志很小，所以日志溢出
   并不是真正的错误情况。

   相反，我们将原始元数据对分为两个元数据对，每个对
   包含一半的条目，由尾指针连接。代替
   增加日志的大小并处理可伸缩性问题
   与较大的日志相关联，我们形成了一个小边界日志的链接列表。
   这是一个折衷，因为这种方法确实使用了更多的存储空间，但是
   改进的可伸缩性的好处。

   尽管写入了两个元数据对，我们仍然可以保持强大的功能
   首先准备新的元数据对，从而在此拆分步骤中恢复弹性，
   然后在提交到原始文件的过程中插入尾指针
   元数据对。

   ```
                      commit C and D, need to split
.----------------.----------------.    .----------------.----------------.
|   revision 1   |   revision 2   | => |   revision 3   |   revision 2   |
|----------------|----------------|    |----------------|----------------|
|       A        |       A'       |    |       A'       |       A'       |
|----------------|----------------|    |----------------|----------------|
|    checksum    |       B'       |    |       B'       |       B'       |
|----------------|----------------|    |----------------|----------------|
|       B        |    checksum    |    |      tail    ---------------------.
|----------------|----------------|    |----------------|----------------| |
|       A'       |       |        |    |    checksum    |                | |
|----------------|       v        |    |----------------|                | |
|    checksum    |                |    |       |        |                | |
|----------------|                |    |       v        |                | |
'----------------'----------------'    '----------------'----------------' |
                                                .----------------.---------'
                                               v                v
                                       .----------------.----------------.
                                       |   revision 1   |   revision 0   |

处理小原木时，还有另一种复杂性。这
垃圾回收的摊销运行时间成本不仅取决于其

考虑两个极端：

1.日志为空，每_n_次更新都会进行垃圾回收
2.日志已满，垃圾收集“每**”更新

显然，我们需要比等待我们的元数据对更积极
满。随着元数据对接近充满，压缩的频率

条目，！[d]动态条目（垃圾期间过时的条目

垃圾收集）。如果我们看一下更新后的摊销运行时复杂性
这个日志我们得到这个公式：

！[cost = n + n（s / d + 1）] [元数据公式1]

如果让！[r]为静态空间与日志大小（以字节为单位）的比率，则我们

！[s = r（大小/ n）] [元数据公式2]

！[d =（1- r）（大小/ n）] [元数据公式3]

将它们替换为！[d]和！[s]为我们提供了一个很好的公式，用于计算成本
给定日志的完整程度，更新条目：

！[cost = n + n（r（大小/ n）/（（1-r）（大小/ n）+ 1））] [元数据公式4]

假设在4 KiB日志中有100个字节的条目，我们可以使用该条目绘制图形
找到可乘费用的大小：

！[元数据对更新成本图] [元数据成本图]

因此，在使用率为50％的情况下，每次更新的平均成本为2倍，而更新率为75％
使用率，我们每次更新的平均费用已经是4倍。

为了避免这种指数增长，请不要等待我们的元数据对
为了完整起见，一旦容量超过50％，我们就会拆分元数据对。我们这样做

给我们带来了_O（1）_的摊销运行时间复杂度。

## CTZ跳过列表

元数据对可提供有效的原子更新，但不幸的是，
存储成本。但是我们只能通过使用
元数据对，用于存储对更密集的写时复制（COW）数据的引用
结构。

[写时复制数据结构] [wikipedia-cow]，也称为纯函数
数据结构是一类数据结构，其中基础
元素是不可变的。更改数据需要创建新的
包含更新数据副本并替换所有引用的元素
并引用了新元素。通常，COW数据的性能
结构取决于更换零件后可以重复使用多少个旧元素
的数据。

littlefs对COW结构有一些要求。他们需要
读写效率高，但最令人沮丧的是它们需要可遍历
具有恒定数量的RAM。值得注意的是，这排除了
[B树] [Wikipedia-B树]，无法使用恒定的RAM进行遍历，并且
[B +-树] [wikipedia-B +-树]，无法使用COW进行更新
操作。

---

所以，我们能做些什么？首先让我们考虑将文件存储在简单的COW中
链表。追加一个块，这是编写文件的基础，这意味着我们
必须更新最后一个块以指向我们的新块。这需要一个牛
操作，这意味着我们需要更新倒数第二个块，然后
倒数第三，依此类推，直到复制完整个文件。

```
A linked-list
.--------.  .--------.  .--------.  .--------.  .--------.  .--------.
| data 0 |->| data 1 |->| data 2 |->| data 4 |->| data 5 |->| data 6 |
|        |  |        |  |        |  |        |  |        |  |        |
|        |  |        |  |        |  |        |  |        |  |        |
'--------'  '--------'  '--------'  '--------'  '--------'  '--------'
```

为了避免在追加期间进行完整复制，我们可以向后存储数据。追加中
块只需要添加新块，而无需其他块
更新。如果我们在中间更新一个块，我们仍然需要复制
后面的块，但可以重用之前的任何块。由于大多数文件写入
是线性的，这种附加设计的赌博是最常见的数据类型
更新。

```
A backwards linked-list
.--------.  .--------.  .--------.  .--------.  .--------.  .--------.
| data 0 |<-| data 1 |<-| data 2 |<-| data 4 |<-| data 5 |<-| data 6 |
|        |  |        |  |        |  |        |  |        |  |        |
|        |  |        |  |        |  |        |  |        |  |        |
'--------'  '--------'  '--------'  '--------'  '--------'  '--------'
```

但是，反向链接列表确实存在一个非常明显的问题。反复进行
在文件_in_上的运行时成本为_O（n＆sup2;）_。二次运行
只是为了读取文件！那是糟糕的。

幸运的是我们可以做得更好。代替单链列表，littlefs
使用通常称为“
[跳过列表] [wikipedia-跳过列表]。但是，与最常见的
跳过列表，littlefs的跳过列表是严格确定性的，围绕一些
count-trailing-zeros（CTZ）指令的有趣属性。

CTZ跳过列表遵循的规则是每个_n_的块，其中_n_
可被2＆zwj; _＆＃739; _整除，该块包含一个指向该块的指针
_n_-2＆zwj; _＆＃739; _。这意味着每个块包含从1到1的任何位置。
log＆＃8322; _n_指针，它们会跳转到
跳过列表。

该名称源于对[CTZ指令] [wikipedia-ctz]的大量使用，
让我们有效地计算两个因子的幂。对于给定区块_n_，
该块包含ctz（_n _）+ 1指针。

```
A backwards CTZ skip-list
.--------.  .--------.  .--------.  .--------.  .--------.  .--------.
| data 0 |<-| data 1 |<-| data 2 |<-| data 3 |<-| data 4 |<-| data 5 |
|        |<-|        |--|        |<-|        |--|        |  |        |
|        |<-|        |--|        |--|        |--|        |  |        |
'--------'  '--------'  '--------'  '--------'  '--------'  '--------'
```

附加的指针使我们可以导航磁盘上的数据结构更多
比单链列表更有效。

考虑从数据块5到数据块1的路径。您可以看到数据块3的情况
被完全跳过：
```

|        |  |        |  |        |<-|        |--|        |  |        |
|        |  |        |  |        |  |        |  |        |  |        |
'--------'  '--------'  '--------'  '--------'  '--------'  '--------'
```

到数据块0的路径甚至更快，只需两次跳转：
```
.--------.  .--------.  .--------.  .--------.  .--------.  .--------.
| data 0 |  | data 1 |  | data 2 |  | data 3 |  | data 4 |<-| data 5 |
|        |  |        |  |        |  |        |  |        |  |        |
|        |<-|        |--|        |--|        |--|        |  |        |
'--------'  '--------'  '--------'  '--------'  '--------'  '--------'
```

我们可以通过查看到任何块的路径来找到运行时复杂性
包含最多指针的块。道路上的每一步都分裂
该块的搜索空间减半，使我们的运行时间为_O（log n）_。
要获得具有最多指针的块，我们可以执行相同的步骤
向后，则将运行时置于_O（2 log n）_ = _O（log n）_。一个有趣的
请注意，如果我们贪婪地选择
指针可以覆盖最远距离而不会超过我们的目标

因此，现在我们有了一个[COW]数据结构，可以便宜地附加一个运行时
_O（1）_的值，并且可以在最坏的情况下_O（n log n）_读取。给定
该运行时也除以我们可以存储在
块，这个费用是相当合理的。

---

这是一个新的数据结构，因此我们仍然有几个问题。是什么
存储开销？指针的数量可以超过块的大小吗？怎么做
我们在元数据对中存储CTZ跳过列表？

为了找到存储开销，我们可以将数据结构视为多个
链表。每个链表跳过的块数是前一个的两倍，
或者从另一个角度来看，每个链表使用的存储量是
以前的。随着我们接近无穷大，存储开销形成了一个几何
系列。解决这个问题告诉我们，平均而言，我们的存储开销仅为
每个块2个指针。

！[lim，n-> inf（（1 / n）sum，i，0-> n（ctz（i）+1））= sum，i，0-> inf（1/2 ^ i）= 2] [ctz-formula1]

因为我们的文件大小受到限制，所以我们用来存储大小的字宽
还解决了我们需要存储在一个指针中的最大指针数
堵塞。如果将指针的开销设置为等于块大小，则得到
以下等式。请注意，较小的块大小（！[B] [bigB]）和较大的块
字宽（！[w]）会导致更多的存储开销。

！[B =（w / 8）ceil（log2（2 ^ w /（B-2w / 8）））] [ctz-formula2]

求解！[B] [bigB]的方程式可为我们提供一些最小的块大小
常用字宽：

1. 32位CTZ跳过列表=>最小块大小为104字节
2. 64位CTZ跳过列表=>最小块大小为448字节

littlefs使用32位字宽，因此我们的块只能使用
小于104个字节的指针。这是一个简单的要求，因为
实际上，大多数块大小从512字节开始。只要我们块大小
大于104个字节，我们可以避免处理所需的额外逻辑
指针溢出。

最后一个问题是我们如何存储CTZ跳过列表？我们需要一个指向
标头，跳过列表的大小，标头的索引以及我们的
头块中的偏移量。但值得注意的是，每个尺寸都映射到一个唯一的
索引+偏移对。因此，从理论上讲，我们只能存储单个指针和大小。

但是，从大小计算索引+偏移对有点
复杂。我们可以从遍历所有块的求和开始
直到我们给定的大小。令！[B] [bigB]为以字节为单位的块大小，！[w]为
字宽（以位为单位），！[n]是跳过列表中块的索引，并且
！[N] [bigN]是文件大小，以字节为单位：

！[N = sum，i，0-> n（B-（w / 8）（ctz（i）+1））] [ctz-公式3]

这工作得很好，但是需要_O（n）_进行计算，这带来了全部
最多读取_O（n＆sup2; log n）_的运行时。幸运的是，总和
不需要接触磁盘，因此实际影响很小。

但是，尽管集成了按位运算，我们实际上可以减少
该方程式为_O（1）_形式。浏览令人惊奇的资源时
我找到了[整数序列在线百科全书（OEIS）] [oeis]
[A001511]与CTZ指令的迭代匹配，
[A005187]，与它的部分求和相匹配。对我而言
令人惊讶的是，这些都是由简单的方程式产生的，使我们得出了
将两个看似无关的按位联系在一起的非直觉属性
指示：

！[sum，i，0-> n（ctz（i）+1）= 2n-popcount（n）] [ctz-公式4]

在哪里：

1. ctz（！[x]）=！[x]中为0的尾随位数
2. popcount（！[x]）=！[x]中的1的位数

这项令人惊讶的特性的初步测试似乎成立了。随着！[n]的临近
无穷大，我们最终得到2个指针的平均开销，这与我们的
早先的假设。在迭代过程中，popcount函数似乎
处理与该平均值的偏差。当然，只是为了确保我写了一个
快速脚本，用于验证所有32位整数的此属性。

现在我们可以代入原始方程式以找到更有效的
文件大小的等式：

！[N = Bn-（w / 8）（2n-popcount（n））] [ctz-formula5]

不幸的是，popcount函数是非内射的，因此我们无法解决此问题
我们的指数方程。但是我们可以做的是求解一个[！n]索引，
大于！[n]，误差由popcount函数的范围限制。
我们可以反复将！[n']代入原始方程式，直到出现错误为止
小于我们的整数分辨率。事实证明，我们只需要
一次执行此替换，这为我们的索引提供了以下公式：

！[n = floor（（N-（w / 8）popcount（N /（B-2w / 8）））/（B-2w / 8））] [ctz-formula6]

现在我们有了索引！[n]，我们可以将其重新插入到上面
方程来找到偏移量。我们遇到整数问题
溢出，但是我们可以通过重新排列等式来避免这种情况：

！[off = N-（B-2w / 8）n-（w / 8）popcount（n）] [ctz-formula7]

我们的解决方案需要大量的数学运算，但是计算机在数学运算方面非常擅长。
现在我们可以在_O（1）_中找到我们的块索引和偏移量，让
我们仅使用指针和大小存储CTZ跳过列表。

CTZ跳过列表为我们提供了一个COW数据结构，该数据结构可轻松在其中进行遍历

这些操作在有限的RAM中工作，仅需要两个字

##块分配器

因此，我们现在有了一个原子的损耗平衡文件系统的框架。小二
块元数据对提供原子更新，而CTZ跳过列表提供
将数据紧凑存储在COW块中。

但是现在我们需要看一下房间中的[elephant]。这些都在哪里
块从何而来？

决定下一步使用哪个块是块分配器的责任。
在文件系统设计中，块分配通常是第二等公民，但是在
COW文件系统的作用变得越来越重要，因为它需要
几乎每一次写入文件系统。

通常，块分配涉及存储在其上的某种空闲列表或位图
用空闲块更新的文件系统。但是，有了力量
弹性，保持这些结构的一致性变得困难。不会
帮助更新这些结构中的任何错误都可能导致丢失块
那是不可能恢复的。

littlefs采取谨慎的态度。与其信任磁盘上的空闲列表，
littlefs依赖于以下事实：磁盘上的文件系统是以下文件的镜像：
磁盘上的可用块。块分配器的操作很像垃圾
以脚本语言收集器，按需扫描未使用的块。

```
          .----.
          |root|
          |    |
          '----'
   v-------'  '-------v
.----.    .    .    .----.
| A  |    .    .    | B  |
|    |    .    .    |    |
'----'    .    .    '----'
.    .    .    .  v--'  '------------v---------v
.    .    .    .----.    .         .----.    .----.
.    .    .    | C  |    .         | D  |    | E  |
.    .    .    |    |    .         |    |    |    |
.    .    .    '----'    .         '----'    '----'
.    .    .    .    .    .         .    .    .    .
.----.----.----.----.----.----.----.----.----.----.----.----.
| A  |    |root| C  | B  |         | D  |    | E  |         |
|    |    |    |    |    |         |    |    |    |         |
'----'----'----'----'----'----'----'----'----'----'----'----'
        ^                   ^    ^                   ^    ^
         '-------------------'----'-------------------'----'-- free blocks
```

尽管这种方法听起来可能很复杂，但是决定不维护自由
该列表大大简化了littlefs的总体设计。与编程不同
语言，我们只需要遍历少数几个数据结构。
而且，区块重新分配几乎与区块分配一样频繁地发生，
简直是小菜一碟。这种“掉在地板上”的策略大大减少了
磁盘数据结构管理的复杂性，尤其是在处理时
高风险错误条件。

我们的块分配器需要有效地找到可用的块。你可以遍历
遍历存储中的每个块，并对照我们的文件系统树检查每个块；
但是，运行时会令人厌恶。我们需要以某种方式收集多个
每个遍历块。

看看现有的设计，一些使用类似“ drop”的较大文件系统
“在地板上”策略将整个存储的位图存储在[RAM]中。
效果很好，因为位图非常紧凑。我们不能使用相同的
此处的策略，因为它违反了我们不变的RAM要求，但我们也许能够
将该想法修改为可行的解决方案。

```
.----.----.----.----.----.----.----.----.----.----.----.----.
| A  |    |root| C  | B  |         | D  |    | E  |         |
|    |    |    |    |    |         |    |    |    |         |
'----'----'----'----'----'----'----'----'----'----'----'----'
  1    0    1    1    1    0    0    1    0    1    0    0
 \---------------------------+----------------------------/
                             v
               bitmap: 0xb94 (0b101110010100)
```

littlefs中的块分配器是磁盘大小的位图和
蛮力遍历。我们会跟踪，而不是存储大小的位图
固定大小的小位图，称为超前缓冲区。在块中
分配时，我们从超前缓冲区中获取块。如果超前缓冲

缓冲。在每次扫描中，我们使用增加的偏移量，将存储空间盘旋为
分配块。

##磨损平衡

块分配器具有次要作用：损耗均衡。

磨损平衡是指将磨损分布到所有块中的过程。
存储以防止文件系统由于以下原因而过早死亡
放在存储区中的一个块上。

littlefs有两种防止磨损的方法：
1.检测并从坏块中恢复
2.在动态块之间平均分配磨损

---

从坏块中恢复实际上与该块没有任何关系
分配器本身。相反，它依赖于文件系统检测的能力
并在发生坏块时将其逐出。

在littlefs中，在写入时检测坏块相当简单。
所有写操作都必须以某种形式的RAM中的数据为源，因此在我们写完之后
写入一个块，我们可以读回数据并验证它是否已写入
正确地。如果我们发现磁盘上的数据与我们拥有的副本不匹配
RAM，发生了写错误，我们很可能有一个坏块。

一旦检测到坏块，就需要从中恢复。在写的情况下
错误，我们在RAM中有一个损坏的数据的副本，所以我们要做的就是
逐出坏块，分配一个新的希望良好的块，然后重复写入
以前失败了。

保留文件系统的“边界写限制复制”（CObW）数据结构。一
CObW数据结构的特性是，在
COW操作。边界写入部分通常由计数器触发，但是
一旦发现问题，没有什么可以阻止我们触发COW操作
堵塞。

错误。从技术上讲这是正确的，因为没有更好的块，但是作为
附加的好处是它还与用户预期的错误条件相匹配
动态调整大小的数据。

---

另一方面，读取错误要复杂得多。我们没有
数据的副本在RAM中徘徊，因此我们需要一种方法来重建
即使原始数据已损坏。一种这样的机制是
[纠错码（ECC）] [wikipedia-ecc]。

ECC是对校验和概念的扩展。像CRC这样的校验和可以
检测到数据中已发生错误，ECC可以检测并实际
更正一些错误。但是，错误数ECC有一个限制
可以检测到：[汉明限界] [维基百科-汉明限界]。作为数量
错误接近汉明界限，我们也许仍然能够检测到错误，但是
无法修复数据。如果我们到了这一点，那么障碍就是
无法恢复。

littlefs本身不提供ECC。块的性质和相对
ECC占用空间大，无法与动态大小的数据一起使用
文件系统，在没有RAM的情况下纠正错误非常复杂，并且ECC更适合
与块设备的几何形状有关。实际上，一些NOR闪存芯片具有额外的功能
用于ECC的存储，许多NAND芯片甚至可以在ECC上计算ECC
芯片本身。

在littlefs中，ECC是完全可选的。可以防止读取错误
通过磨损均衡主动地进行。但请务必注意，可以使用ECC
在块设备级别适度延长设备寿命。littlefs
尊重块设备报告的任何错误，从而允许块设备执行以下操作：
提供其他主动错误检测。

---

为了避免读取错误，我们需要积极主动，而不是被动被动
有写错误。

一种方法是检测块中的错误数何时超过
某个阈值，但仍然可以恢复。借助ECC，我们可以在写入时执行此操作
时间，并将错误视为写入错误，在致命之前驱逐该块
读取错误有机会发展。

另一种更通用的策略是主动分配磨损
存储中的所有块，希望没有单个块在
其余的存储即将用完。这就是所谓的
磨损均衡。

通常，损耗平衡算法属于以下两类之一：

1. [动态磨损平衡] [wikipedia-动态磨损平衡]
   在“动态”块上分配磨损。可以通过以下方式完成
   仅考虑未使用的块。

2. [静态损耗均衡] [wikipedia-static-wear-leveling]，其中
   在“动态”和“静态”块上分配磨损。为了使这项工作，
   我们需要考虑所有块，包括已经包含数据的块。

作为代码大小和复杂性的折衷，littlefs（当前）仅提供
动态磨损平衡。这是尽力而为的解决方案。磨损不均匀
完美，但它分布在空闲块之间，并极大地扩展了
设备的寿命。

最重要的是，littlefs使用统计损耗均衡算法。这是什么
意味着我们不主动跟踪磨损，而是依靠制服
整个存储中的磨损分布以近似动态磨损均衡
算法。尽管名字很长，但这实际上是对动态的简化。
磨损均衡。

磨损的均匀分布将留给块分配器，由块分配器
在两个部分中创建均匀分布。最简单的部分是当设备
供电，在这种情况下，我们以线性方式分配块，围绕设备运行。
最难的部分是设备断电时该怎么办。我们不能只是
在存储开始时重新启动分配器，因为这会增加损耗。
取而代之的是，每次安装
文件系统。只要此随机偏移量是均匀的，则组合分配
图案也是均匀分布的。

！[累积磨损分布图] [磨损分布图]

最初，这种磨损平衡的方法看起来很困难
依赖于与功率无关的随机数生成器，该生成器必须返回
每次启动时都有不同的随机数。但是，文件系统位于
相对独特的情况是，它位于大量
熵在整个功率损耗中持续存在。

我们实际上可以使用磁盘上的数据直接驱动我们的随机数
发电机。实际上，这是通过对每个校验和进行异或来实现的
元数据对，已经计算出该元数据对以获取并挂载文件系统。

```

           .|metadata| |                                ^
           ||        | +-> crc ----------------------> xor

           v       crc               crc      v        crc
      .--------. \  ^   .--------. \  ^   .--------. \  ^

一起，坏块检测和动态磨损平衡可提供最大的努力

littlefs的损耗均衡算法提供了一项关键功能：您可以增加
仅通过增加存储大小即可延长设备的使用寿命。还有更多

每个目录本身都是元数据对的链接列表。这让我们
在每个目录中存储无限数量的文件，我们不需要
担心无界日志的运行时复杂性。我们可以存储其他
我们的元数据对中的目录指针，这给了我们目录树
就像您在其他文件系统上找到的一样。

```
            .--------.
           .| root   |
           ||        |
           ||        |
           |'--------'
           '---|--|-'
            .-'    '-------------------------.
           v                                  v
      .--------.        .--------.        .--------.
     .| dir A  |------->| dir A  |       .| dir B  |
     ||        |       ||        |       ||        |
     ||        |       ||        |       ||        |
     |'--------'       |'--------'       |'--------'
     '---|--|-'        '----|---'        '---|--|-'
      .-'    '-.            |             .-'    '-.
     v          v           v            v          v
.--------.  .--------.  .--------.  .--------.  .--------.
| file C |  | file D |  | file E |  | file F |  | file G |
|        |  |        |  |        |  |        |  |        |
|        |  |        |  |        |  |        |  |        |
'--------'  '--------'  '--------'  '--------'  '--------'
```

同样，主要的复杂之处是遍历常量
[内存]。目录树是一棵树，不幸的事实是您不能
遍历具有恒定RAM的树。

幸运的是，我们树的元素是元数据对，因此与CTZ不同
跳过列表，我们不仅限于严格的COW操作。我们可以做的一件事是
通过我们的树线程化链接列表，显式启用廉价遍历
在整个文件系统上。

```
            .--------.
           .| root   |-.
           ||        | |
   .-------||        |-'
   |       |'--------'
   |       '---|--|-'
   |        .-'    '-------------------------.
   |       v                                  v
   |  .--------.        .--------.        .--------.
   '->| dir A  |------->| dir A  |------->| dir B  |
     ||        |       ||        |       ||        |
     ||        |       ||        |       ||        |
     |'--------'       |'--------'       |'--------'
     '---|--|-'        '----|---'        '---|--|-'
      .-'    '-.            |             .-'    '-.
     v          v           v            v          v
.--------.  .--------.  .--------.  .--------.  .--------.
| file C |  | file D |  | file E |  | file F |  | file G |
|        |  |        |  |        |  |        |  |        |
|        |  |        |  |        |  |        |  |        |
'--------'  '--------'  '--------'  '--------'  '--------'
```

不幸的是，不坚持纯COW操作会带来一些问题。现在，
每当我们要操作目录树时，都需要使用多个指针
更新。如果您熟悉设计原子数据结构，则应该
掀起一堆红旗。

要解决此问题，我们的线程化链表还有一些余地。代替
仅包含在我们的文件系统中找到的元数据对，允许
包含因断电而没有父级的元数据对。这些都是
称为孤立元数据对。

在有孤儿的情况下，我们可以建立具有抗功耗能力的运营
维护带有链表进行遍历的文件系统树。

##移动问题

我们还有最后一个挑战：搬家问题。短语问题很简单：

您如何在两个目录之间原子地移动文件？

在littlefs中，我们可以原子地提交到目录，但不能创建
跨多个目录的原子提交。文件系统必须去
通过至少两个不同的状态来完成移动。

更糟糕的是，文件移动是同步的一种常见形式
文件系统。作为旨在减少功耗的文件系统，重要的是
原子向右移动。

所以，我们能做些什么？

-我们绝对不能只因掉电而导致文件重复或丢失。
  这很容易破坏用户的代码，并且只会在极端情况下暴露自己
  案件。我们只能对线程化的链表感到懒惰，因为
  它不是面向用户的，我们可以在内部处理极端情况。

-一些文件系统将COW操作传播到树上，直到有一个公共父级
  被发现。不幸的是，这与我们的线程树交互性很差，并且
  带回了磨损向上传播的问题。

-在littlefs的先前版本中，我们尝试通过以下方式解决此问题：
  在源和目的地之间来回移动，标记和取消标记
  将文件视为移动文件，以便从用户角度使移动成为原子文件。
  这行得通，但效果不佳。寻找失败的举动既昂贵又需要
  每个文件的唯一标识符。

最后，解决搬家问题需要创建一种新的机制
在多个元数据对之间共享知识。在littlefs中，这导致了
引入一种称为“全局状态”的机制。

---

全局状态是可以从_any_元数据更新的一小部分状态
一对。结合全局状态和元数据对的更新多个功能的能力
一次提交中的条目为我们提供了一种用于构造复杂原子的强大工具
操作。

全球国家如何运作？

全局状态以分布在元数据中的一组增量形式存在
对在文件系统中。实际的全球状态可以由这些构成
通过将文件系统中的所有增量异或在一起来实现增量。

```
 .--------.  .--------.  .--------.  .--------.  .--------.
.|        |->| gdelta |->|        |->| gdelta |->| gdelta |
||        | || 0x23   | ||        | || 0xff   | || 0xce   |
||        | ||        | ||        | ||        | ||        |
|'--------' |'--------' |'--------' |'--------' |'--------'
'--------'  '----|---'  '--------'  '----|---'  '----|---'
                 v                       v           v
       0x00 --> xor ------------------> xor ------> xor --> gstate 0x12
```

要从元数据对更新全局状态，我们采用全局状态
通过我们的更改和元数据中的任何现有增量了解并异或
一对。将此新的增量提交给元数据对，会将更改提交到
文件系统的全局状态。

```
 .--------.  .--------.  .--------.  .--------.  .--------.
.|        |->| gdelta |->|        |->| gdelta |->| gdelta |
||        | || 0x23   | ||        | || 0xff   | || 0xce   |
||        | ||        | ||        | ||        | ||        |
|'--------' |'--------' |'--------' |'--------' |'--------'
'--------'  '----|---'  '--------'  '--|---|-'  '----|---'
                 v                     v   |         v
       0x00 --> xor ----------------> xor -|------> xor --> gstate = 0x12
                                           |                          |
                                           |                          |
change gstate to 0xab --> xor <------------|--------------------------'
=>                         |               v
                           '------------> xor
                                           |
                                           v
 .--------.  .--------.  .--------.  .--------.  .--------.
.|        |->| gdelta |->|        |->| gdelta |->| gdelta |
||        | || 0x23   | ||        | || 0x46   | || 0xce   |
||        | ||        | ||        | ||        | ||        |
|'--------' |'--------' |'--------' |'--------' |'--------'
'--------'  '----|---'  '--------'  '----|---'  '----|---'
                 v                       v           v
       0x00 --> xor ------------------> xor ------> xor --> gstate = 0xab
```

为了提高效率，我们始终在RAM中保留全局状态的副本。我们
只需要遍历我们的元数据对并在以下情况下建立全局状态
文件系统已挂载。

您可能已经注意到，全局状态非常昂贵。我们保留一份副本
RAM和无限数量的元数据对中的增量。即使重设
全局状态恢复到其初始值，我们无法轻松清理
磁盘。因此，保持全球规模非常重要
国家有界并且非常小。但是，即使预算很严格，全球
状态是非常宝贵的。

---

现在我们可以解决搬家问题。我们可以创建描述我们的全球状态
通过创建新文件进行原子移动，我们可以清除此移动
删除旧文件，实现原子状态。

```
               .--------.    gstate = no move
              .| root   |-.
              ||        | |
.-------------||        |-'
|             |'--------'
|             '--|-|-|-'
|        .------'  |  '-------.
|       v          v           v
|  .--------.  .--------.  .--------.
'->| dir A  |->| dir B  |->| dir C  |
  ||        | ||        | ||        |
  ||        | ||        | ||        |
  |'--------' |'--------' |'--------'
  '----|---'  '--------'  '--------'
       v
   .--------.
   | file D |
   |        |
   |        |
   '--------'

begin move, add reference in dir C, change gstate to have move
=>
               .--------.    gstate = moving file D in dir A (m1)
              .| root   |-.
              ||        | |
.-------------||        |-'
|             |'--------'
|             '--|-|-|-'
|        .------'  |  '-------.
|       v          v           v
|  .--------.  .--------.  .--------.
'->| dir A  |->| dir B  |->| dir C  |
  ||        | ||        | || gdelta |
  ||        | ||        | || =m1    |
  |'--------' |'--------' |'--------'
  '----|---'  '--------'  '----|---'
       |     .----------------'
       v    v
     .--------.
     | file D |
     |        |
     |        |
     '--------'

complete move, remove reference in dir A, change gstate to no move
=>
               .--------.    gstate = no move (m1^~m1)
              .| root   |-.
              ||        | |
.-------------||        |-'
|             |'--------'
|             '--|-|-|-'
|        .------'  |  '-------.
|       v          v           v
|  .--------.  .--------.  .--------.
'->| dir A  |->| dir B  |->| dir C  |
  || gdelta | ||        | || gdelta |
  || =~m1   | ||        | || =m1    |
  |'--------' |'--------' |'--------'
  '--------'  '--------'  '----|---'
                               v
                           .--------.
                           | file D |
                           |        |
                           |        |
                           '--------'
```


如果在挂载期间建立了全局状态之后，我们找到了信息
描述正在进行的动作，我们知道我们在移动过程中失去了动力，并且文件
在源目录和目标目录中都重复。如果发生这种情况
我们可以使用全局状态下的信息来解决此举
其中一个文件。

```
                 .--------.    gstate = moving file D in dir A (m1)
                .| root   |-.             ^
                ||        |------------> xor
.---------------||        |-'             ^
|               |'--------'               |
|               '--|-|-|-'                |
|        .--------'  |  '---------.       |
|       |            |             |      |
|       |     .----------> xor --------> xor
|       v     |      v      ^      v      ^
|  .--------. |  .--------. |  .--------. |
'->| dir A  |-|->| dir B  |-|->| dir C  | |
  ||        |-' ||        |-' || gdelta |-'
  ||        |   ||        |   || =m1    |
  |'--------'   |'--------'   |'--------'
  '----|---'    '--------'    '----|---'
       |     .---------------------'
       v    v
     .--------.
     | file D |
     |        |
     |        |
     '--------'
```

我们也可以像移动文件一样移动目录。有螺纹
要考虑的链表，但保持线程化链表不变
很好，因为顺序并不重要。

```
               .--------.    gstate = no move (m1^~m1)
              .| root   |-.
              ||        | |
.-------------||        |-'
|             |'--------'
|             '--|-|-|-'
|        .------'  |  '-------.
|       v          v           v

  || gdelta | ||        | || gdelta |
  || =~m1   | ||        | || =m1    |
  |'--------' |'--------' |'--------'
  '--------'  '--------'  '----|---'
                               v
                           .--------.
                           | file D |
                           |        |
                           |        |
                           '--------'

begin move, add reference in dir C, change gstate to have move
=>
                .--------.    gstate = moving dir B in root (m1^~m1^m2)
               .| root   |-.
               ||        | |
.--------------||        |-'
|              |'--------'
|              '--|-|-|-'
|        .-------'  |  '----------.
|       v           |              v
|  .--------.       |          .--------.
'->| dir A  |-.     |       .->| dir C  |
  || gdelta | |     |       | || gdelta |
  || =~m1   | |     |       | || =m1^m2 |
  |'--------' |     |       | |'--------'
  '--------'  |     |       | '---|--|-'
              |     |    .-------'   |
              |     v   v   |        v
              |  .--------. |    .--------.
              '->| dir B  |-'    | file D |
                ||        |      |        |
                ||        |      |        |
                |'--------'      '--------'
                '--------'

complete move, remove reference in root, change gstate to no move
=>
             .--------.    gstate = no move (m1^~m1^m2^~m2)
            .| root   |-.
            || gdelta | |
.-----------|| =~m2   |-'
|           |'--------'
|           '---|--|-'
|        .-----'    '-----.
|       v                  v
|  .--------.          .--------.
'->| dir A  |-.     .->| dir C  |
  || gdelta | |     | || gdelta |
  || =~m1   | |     '-|| =m1^m2 |-------.
  |'--------' |       |'--------'       |
  '--------'  |       '---|--|-'        |
              |        .-'    '-.       |
              |       v          v      |
              |  .--------.  .--------. |
              '->| dir B  |--| file D |-'
                ||        |  |        |
                ||        |  |        |
                |'--------'  '--------'
                '--------'
```

全局状态为我们提供了一个强大的工具，可以用来解决搬迁问题。
结果出乎意料的是性能出色，只需要最少的数量
状态和使用相同数量的提交作为幼稚的举动。此外，
全局状态为我们提供了一些持久状态，我们可以将其用于其他一些状态
小改进。
